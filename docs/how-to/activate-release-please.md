## What release-please does

On every push to `main`, release-please:

1. Reads [Conventional Commits](https://www.conventionalcommits.org)
   since the last `vX.Y.Z` tag.
2. Computes the next semver version (`feat: ...` → minor bump,
   `fix: ...` / `perf: ...` → patch, `feat!: ...` → major).
3. Writes the new entry at the top of `CHANGELOG.md` grouped by
   section (Features, Bug fixes, Performance, etc. — see
   `changelog-sections` in `config/release-please-config.json`).
4. Bumps the version in `pom.xml` (via `extra-files` config).
5. Opens a "Release PR" titled something like `chore(main): release
   0.2.0`. **Merging this PR** is what actually releases — it tags
   `v0.2.0` and creates a GitLab Release.

If no new conventional commits since last release → no PR opened.
Zero noise.

## Wiring — 5 minutes

### Step 1 — Create a GitLab Personal Access Token

1. GitLab → User avatar (top right) → **Preferences** → **Access Tokens**
   (or directly <https://gitlab.com/-/user_settings/personal_access_tokens>).
2. Click **Add new token**.
3. **Name**: `mirador-release-please`.
4. **Expiration**: 1 year (renewable; release-please gracefully skips
   if the token expires — it doesn't break MAIN pipelines).
5. **Scopes**: check **`api`** + **`write_repository`**. No more, no less.
6. **Create personal access token** → COPY THE TOKEN IMMEDIATELY. You
   won't see it again.

### Step 2 — Add the token as a CI/CD variable

1. <https://gitlab.com/mirador1/mirador-service/-/settings/ci_cd>
   → **Variables** section → click **Expand** → **Add variable**.
2. Fill:
   - **Key**: `RELEASE_PLEASE_TOKEN` (exactly, case-sensitive).
   - **Value**: paste the token from step 1.
   - **Type**: Variable (not File).
   - **Environment scope**: `*` (all).
   - **Visibility**: **Masked** (important — hides the token in CI logs).
   - **Protect variable**: ✅ (only available to protected branches,
     and `main` is protected).
   - **Expand variable reference**: unchecked (no shell-expansion).
3. **Add variable**.

### Step 3 — Trigger the first run

release-please's CI job fires on every `main` push. Trigger a no-op:

```bash
# From the mirador-service repo, on dev
git commit --allow-empty --signoff -m "chore: trigger initial release-please run"
git push origin dev
# Create MR + merge per usual flow
```

Or (simpler) — at the next real feature merge, release-please will
fire automatically.

### Step 4 — Verify

Once the main pipeline finishes:

1. Check <https://gitlab.com/mirador1/mirador-service/-/merge_requests>
   for an open MR titled `chore(main): release X.Y.Z` (release-please
   names its PR with this convention).
2. If the MR is there → success. Review the diff:
   - `CHANGELOG.md` gets a new entry at the top grouping all
     conventional commits since the last tag.
   - `pom.xml` version bumped.
   - `.release-please-manifest.json` version bumped.
3. Merge the release MR. release-please creates a `vX.Y.Z` tag + a
   GitLab Release page.

## How it interacts with our tag conventions

Mirador uses **two parallel tag sets**:

| Tag pattern | Who creates | Meaning |
|---|---|---|
| `stable-vX.Y.Z` | Claude / human | "Stability checkpoint" — any green main can be tagged |
| `vX.Y.Z` | release-please | semver release — only when feat/fix commits land |

They can coexist. `stable-v1.0.11` is the human-visible stability
promise; release-please's first tag will probably be `v0.2.0` (from
the accumulated feat commits) and it's the automation-visible release.

A reviewer looking at
<https://gitlab.com/mirador1/mirador-service/-/tags>
sees both lines running in parallel.

## Troubleshooting

**"No release-please MR after main push"**:

- Check the `release-please` job status on the main pipeline. If it's
  skipped, the `RELEASE_PLEASE_TOKEN` CI variable isn't set (or the
  rule gating doesn't match). Re-verify step 2 above.
- Check job logs for `403 Forbidden`: the token lacks `api` or
  `write_repository` scope. Regenerate with both.
- Check job logs for `Resource not found`: the token belongs to a user
  without access to the repo. Ensure it's YOUR PAT, not a bot PAT
  without repo membership.

**"release-please MR opens but is empty"**:

- No conventional commits since the last tag. Non-feat / non-fix
  commits (like `chore:`, `ci:`) don't bump version by default. Check
  the `changelog-sections` in `config/release-please-config.json` —
  any section with `"hidden": false` contributes to the release PR.
- Quick fix: run `git log stable-v1.0.11..HEAD --oneline` and look for
  `feat:` / `fix:` / `perf:` / `docs:` prefixes. If none, no release
  needed — that's correct behaviour.

**"CHANGELOG.md has a merge conflict on release-please MR"**:

- Someone (you, or a pre-release edit) modified `CHANGELOG.md` in the
  meantime. Either rebase the release MR on main, or close + let
  release-please regenerate on the next main push.

## Rotation

Every ~12 months (token expiry approaching):

1. Generate a new PAT with the same scopes + same expiry policy.
2. Update the `RELEASE_PLEASE_TOKEN` CI variable with the new value.
3. Revoke the old PAT from the GitLab Access Tokens page.

Release-please is dormant during the rotation window — no breakage,
just no new release PRs until the variable is current again.
